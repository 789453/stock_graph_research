# README FOR TRAE

## 0. 你的角色

你不是在开发生产系统。  
你是在协助完成一项**研究优先**的工程任务：先把一张真实、可复现、可诊断的语义近邻图做出来，再决定下一步要不要继续。

你的职责是执行清晰规格，不是替研究者扩大问题。

## 1. 每次新窗口启动时，必须先读的文档

按顺序阅读：

1. `RESEARCH_CONSTITUTION.md`
2. `DATA_CONTRACTS.md`
3. `PHASE1_SEMANTIC_GRAPH_ENGINEERING_SPEC.md`
4. 当前项目根目录下的 `PROJECT_STATE.md`
5. 当前任务卡 `tasks/TASK_*.md`（如果存在）

未读完以上文档前，不要开始改代码。

## 2. 第一轮绝对边界

### 允许做

- 使用真实 `application_scenarios_json-all.npy`
- 使用与其配套的 `application_scenarios_json-all.meta.json`
- 使用同一数据集的 `records-all.parquet` 进行 `record_id -> stock_code` 对齐
- 使用 FAISS GPU 构建 kNN
- 缓存节点表、邻居矩阵、边表、诊断统计、可视化布局
- 从缓存绘制图表
- 用当前 `stock_sw_member` 做**解释性标签诊断**
- 对 2010-01-01 至 2026-04 数据做**行情覆盖率普查**

### 不允许做

- 不允许使用 mock 语义向量
- 不允许使用 TF-IDF 代替真实 embedding
- 不允许使用 PCA 降维后的向量代替真实 embedding 构图
- 不允许在第一轮加入图融合、图平滑、因子、回测、模型训练
- 不允许把当前申万成分当成 2010—2026 的历史行业真值
- 不允许为了“让测试通过”偷偷换数据源
- 不允许把研究结果阈值写成工程测试，例如“行业纯度必须大于某个数”

## 3. 修改代码前必须先回答的五个问题

1. 本次任务验证的研究问题是什么？
2. 本次任务读入的真实输入文件是什么？
3. 本次任务产出的落盘文件是什么？
4. 本次任务的失败条件是什么？
5. 本次任务完成后，`PROJECT_STATE.md` 要增加什么？

如果答不清，不要直接写代码。

## 4. 工作方式

- 一次只做一张任务卡。
- 每次修改前先列出将修改的文件。
- 每次修改后：
  1. 运行该任务最小必要测试；
  2. 生成或检查该任务的缓存产物；
  3. 更新 `PROJECT_STATE.md`；
  4. 记录发现的偏差、未解决问题、下一步建议。
- 如果代码与文档冲突，先报告冲突，不要私自改研究边界。
- 如果发现缺文件或数据不符合契约，直接失败并说明，不要自动 fallback。

## 5. 推荐给 Trae 的启动提示词

请先阅读 `docs/README_FOR_TRAE.md`、`docs/RESEARCH_CONSTITUTION.md`、`docs/DATA_CONTRACTS.md`、`docs/PHASE1_SEMANTIC_GRAPH_ENGINEERING_SPEC.md` 和根目录的 `PROJECT_STATE.md`。  
本轮只执行当前任务卡，不扩大范围。  
严禁使用 mock、TF-IDF、PCA 替代真实 1024 维语义向量。  
如果发现输入数据、缓存或代码与文档不一致，先报告，不要自作主张修成另一个项目。  
完成后请给出：修改文件、真实输入、真实输出、运行命令、测试结果、缓存路径、仍未解决的问题，并更新 `PROJECT_STATE.md`。

## 6. 研究者审阅 AI 输出时的最低检查

- 它有没有偷偷引入 fallback？
- 它有没有从缓存绘图，还是又把上游重新算了一遍？
- 它有没有把“描述性结果”误写成“工程验收阈值”？
- 它有没有把当前申万成分当作历史标签？
- 它有没有在本轮之外擅自加入聚类、平滑、因子、回测？
- 它有没有更新 `PROJECT_STATE.md`？
